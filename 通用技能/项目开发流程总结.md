# 项目开发流程总结

> 适用于纯前端 / 小工具 / 单文件应用类项目的通用开发方法论。

---

## 阶段零：项目前期探测（禁止跳过）

动手写代码之前，必须先提问澄清需求。即使请求看起来详细，早期澄清也能避免后续返工。

**核心原则：一次只问一个问题，并提供编号选项。** 等用户回复后再问下一个，绝不批量发问。

示例：
> 视觉风格你喜欢哪种？
> 1. 像素复古风（推荐）
> 2. 极简扁平风
> 3. 你有其他想法吗？

### 第一轮：项目基础（先问）
1. 用一两句话描述你的项目
2. 目标用户是谁？解决什么问题？
3. 有没有参考或灵感来源？喜欢/不喜欢哪里？

### 第二轮：范围和功能（第一轮后再问）
4. 2-4 个必须有的功能？哪些可以后续再加？
5. 需要登录/账号吗？
6. 数据需要跨会话/跨设备持久化吗？
7. 预期使用规模？（单用户、小群体、公开）
8. 有哪些页面/屏幕？列出每个独立视图。
9. 页面间怎么连接？画出页面层级（如：首页 → 详情，tab 栏，弹窗）。

### 第三轮：技术偏好（按需提问）
10. 技术栈偏好？（HTML/CSS/JS、React、Vue 等）
11. 设计风格？（极简、趣味、专业、暗色模式）
12. 平台/浏览器限制？（离线、仅移动端、微信小程序）
13. 部署目标？（GitHub Pages、云平台、自建服务器）

### 第四轮：边界条件（按需提问）
14. 出错时怎么处理？（无网络、API 失败、空数据）
15. 需要多语言或无障碍支持吗？
16. 需要第三方 API 吗？（AI、地图、支付、统计）

### 何时停止提问
- 用户明确说"开始做吧"或"你定就行"
- MVP 范围已经足够清晰
- 至少完成第一、二轮

无法回答的问题，做出合理假设、明确告知，然后继续。

### 阶段零产出
生成 `docs/需求概述.md`，总结用户需求、目标用户、MVP 范围。

## 阶段一：需求文档化

### 1.1 产品需求文档（PRD）
写清以下内容：
- **产品定位**：一句话说清楚这是什么
- **目标用户**：谁在用、什么场景下用
- **核心功能模块**：按场景/页面拆解，每个模块列出关键交互
- **非功能性要求**：性能、平台、离线/在线等约束

### 1.2 数据结构设计
- 列出所有核心数据实体
- 每个实体定义字段名、类型、含义
- 明确实体之间的关系（一对一、一对多）
- 示例数据要写在文档里，方便后续直接拷贝到代码中

### 1.3 算法/规则说明
- 核心计算逻辑单独成文
- 用伪代码或自然语言描述计算步骤
- 标注边界情况和特殊处理

### 1.4 视觉设计规范
- 配色方案（主色、辅色、背景色、文本色）
- 字体与字号层级
- 组件样式约定（按钮、面板、弹窗的圆角/阴影/间距）

### 阶段一产出（保存到 docs/ 文件夹）
- `docs/PRD.md` — 产品需求文档
- `docs/数据结构设计.md` — 数据实体、字段、关系、样本数据
- `docs/UI设计规范.md` — 配色、字体、组件样式
- `docs/大纲.md` — 项目结构、页面树、实现顺序

---

## 阶段二：技术选型与架构

### 2.1 技术选型
根据需求选择合适的技术栈：
- **纯前端单文件**：HTML + Canvas + 原生 JS（适合轻量级、离线场景）
- **框架应用**：React / Vue + UI 库（适合复杂交互）
- **全栈应用**：前后端分离 + 数据库（适合需要持久化、多用户场景）

### 2.2 架构设计
- 画出模块关系图
- 定义状态管理方案（全局 state 结构）
- 定义场景/路由切换逻辑
- 确定持久化方案（localStorage / IndexedDB / 后端 API）

### 阶段二产出（保存到 docs/ 文件夹）
- `docs/技术架构.md` — 技术栈决策、模块图、状态管理、持久化方案
- `docs/项目结构.md` — 目录树、文件命名、模块职责

---

## 阶段三：编码实现

### 3.1 实现顺序（推荐）
1. **骨架先行**：先搭 HTML/CSS 框架 + Canvas 画布初始化
2. **状态管理**：定义全局 state 对象和场景切换函数
3. **数据注入**：将文档中的核心数据转为 JS 常量
4. **逐场景实现**：启动页 → 主界面 → 核心玩法 → 结算页面
5. **弹窗/辅助模块**：设置、档案、提示等叠加层
6. **持久化**：存档/读档逻辑
7. **音效/动画**：最后抛光

### 3.2 编码原则
- 每个函数职责单一
- 弹窗类组件独立渲染，不与具体场景耦合
- UI 交互通过统一的注册机制，避免散落的点击事件
- 数据与逻辑分离：常量数据放配置区，逻辑放函数区
- **防御性编程**：关键流程全程 `try-catch` 包裹，防止单点故障导致页面卡死

### 阶段三产出（保存到 docs/ 文件夹）
- `docs/开发日志.md` — 随实现过程记录关键决策和问题解决
- `docs/环境配置.md` — 开发环境、依赖、启动方式

---

## 阶段四：测试与修复

### 4.1 功能测试
按场景逐一跑通完整流程：
- 正常路径（完整走通全部页面）
- 异常路径（失败、中途退出、存档恢复）
- 边界情况（空数据、极限数值）

### 4.2 代码审查清单
- 检查 **JS 语法错误**
- 检查 **中文引号/特殊字符**（容易在字符串拼接时出问题）
- 搜索 **console.log / debugger** 残留
- 检查 **重复 ID / 重复调用**
- 检查 **废弃代码**（之前修改留下的冗余逻辑）
- 检查 **页面生命周期陷阱**：跳转/刷新时自动保存可能回写已清除的数据

### 4.3 常见陷阱

| 问题 | 表现 | 预防 |
|------|------|------|
| 弹窗与场景耦合 | 弹窗关闭后跳回错误页面 | 弹窗独立渲染，不随场景切换 |
| ID 重复 | 后注册的按钮不响应 | 每个交互元素用唯一 ID |
| 状态切换遗漏 | 切换场景后上一场景的 UI 残留 | 切换时清理中间状态 |
| 自动保存回写 | 重置数据后又出现旧数据 | 重置前设标记位，保存函数检测到标记后跳过 |
| CDN 缓存 | 部署后页面不更新 | 文件名带内容哈希，每次构建生成新文件名 |

### 阶段四产出（保存到 docs/ 文件夹）
- `docs/测试报告.md` — 测试用例、发现的 Bug、审查清单结果

## 阶段五：上线发布

### 5.1 项目整理
1. 补充 `.gitignore`（排除本地缓存、编辑器配置）
2. 写 `README.md`（项目介绍 + 快速开始 + 文档导航）
3. 确保所有文档与代码同步

### 5.2 Git 与 GitHub
```bash
git init                      # 初始化仓库
git add -A                    # 暂存所有文件
git commit -m "提交信息"        # 首次提交
git branch -M main            # 统一分支名
git remote add origin <URL>   # 关联远程仓库
git push -u origin main       # 推送到 GitHub
```

### 5.3 发布后的可选设置
- 添加仓库描述和 Topics（方便搜索）
- 如果是可部署的 Web 应用：启用 GitHub Pages 或云静态托管

### 阶段五产出
- `README.md` — 项目介绍、快速开始、功能列表、文档索引
- `CHANGELOG.md` — 版本变更记录
- `.gitignore` — 排除本地缓存和编辑器配置
- `docs/部署文档.md` — 构建、部署步骤、CDN 缓存策略
- `docs/常见问题.md` — 开发、构建、部署中常见问题与解决方案

## 文档交付规则

生成文档时：
- **完整内容直接写入文件**，不要在对话中贴全文
- **对话中只给 2-3 句简短总结**，说明文档内容和用途
- 这样保持对话简洁、节奏快

## 流程速查

```
需求概述 → PRD + 大纲 + 数据设计 + UI规范
    ↓
技术选型 + 技术架构 + 项目结构
    ↓
骨架 → 状态管理 → 数据注入
    ↓
场景实现 → 辅助模块 → 持久化
    ↓  (同步产出：开发日志 + 环境配置)
全流程测试 → 代码审查 → Bug修复
    ↓  (产出：测试报告)
README + CHANGELOG + .gitignore + 部署文档 + 常见问题 → Git提交 → 发布
```

---

## 适用场景

此流程适用于：
- 单文件/轻量级 Web 应用
- 小游戏/互动工具
- 原型验证项目
- 个人作品/演示 Demo

不适用于（需要额外步骤）：
- 多人员协作项目 → 需要分支策略 + Code Review 流程
- 后端服务项目 → 需要 API 设计 + 数据库迁移 + 部署运维
- 移动原生应用 → 需要平台适配 + 应用商店审核
