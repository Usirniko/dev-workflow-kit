## 四、遇到的主要困难及解决方案

### 4.1 数据迁移兼容

**困难**：日记系统从 v1 → v2 → v3 迭代时，数据结构发生根本性变化（数组→对象），需要保证存量用户数据不丢失。

**解决方案**：
- 在 `diary.js` 中实现两条迁移路径：
  - `migrateFromOld()` — 将旧 `penguin_diary_entries` 数组格式转为新格式
  - `migrateFromV2()` — 将 `penguin_diary_data` 转为 `penguin_diary_data_v3`
- 迁移过程静默处理异常，失败时仅 `console.warn`，不影响主流程
- 存储键名加版本后缀（如 `_v3`），彻底隔离新旧数据

**涉及文件**：`js/diary.js`（第 38 行自动调用迁移）

---

### 4.2 localStorage 存储空间溢出

**困难**：聊天记录和知识库上传文件（PDF/DOCX 内容）可能导致 `QuotaExceededError`。

**解决方案**：
- `storage.js` 实现安全读写封装：
  - 写入前检查异常，捕获 `QuotaExceededError`
  - 自动调用 `cleanupOldChats()` 清理旧聊天记录（保留最近 30 条）
  - 读取时捕获 JSON 解析异常，自动删除损坏数据并返回默认值
- 聊天历史限制上下文窗口为最近 20 轮
- 知识库上传限制单文件 ≤ 10MB

**涉及文件**：`js/storage.js`

---

### 4.3 Canvas 渲染时机问题

**困难**：企鹅 Canvas 在页面初始隐藏状态（`display:none`）时，容器宽度为 0，导致企鹅比例计算错误。

**解决方案**：
- 在 `penguin.js` 中添加兜底逻辑：容器宽度为 0 时使用 Canvas 自身 `width` 属性或默认值
- 企鹅重渲染函数在每次页面显示时重新计算尺寸
- 帧跳过机制（`skipFrames`）在动画速度控制中用于慢速设备

**涉及文件**：`js/penguin.js`（第 29 行兜底逻辑）

---

### 4.4 AI 对话的离线兜底

**困难**：用户未配置 API Key 或网络故障时，聊天功能不能完全不可用。

**解决方案**：
- 实现**三级匹配策略**：
  1. 离线问答库匹配（`OfflineQA.match`，score ≥ 0.55 直接返回）
  2. DeepSeek API 调用（最多重试 2 次，指数退避）
  3. 三级兜底回复（完全离线时返回友好提示）
- 离线问答库 41 条预设问答，按时段/年级匹配
- API 异常分类处理：401不重试、429等待后重试、5xx重试2次、超时15s

**涉及文件**：`js/chat.js`、`js/offline-qa.js`

---

### 4.5 活动系统防刷设计

**困难**：活动系统需要防止用户通过刷关键词无限制获得奖励。

**解决方案**：
- 4 条防刷规则：
  1. 一次性活动 ID 存入 `completedActivities` 数组，已存在则跳过
  2. 可重复活动每日限 3 次（按 `日期+活动ID` 计数）
  3. 相同关键词 5 分钟内不重复触发（`sessionStorage` 冷却记录）
  4. 消息长度 ≥ 5 字才算有效对话（过滤"嗯""哦""好"）
- 每日活动计数自动清理 7 天前的旧记录

**涉及文件**：`js/activity.js`

---

### 4.6 UI 接口向后兼容

**困难**：UI 模块接口在迭代中发生变化，但部分旧调用保留。

**解决方案**：
- 在 `ui.js` 中保留兼容旧方法（标有 `// 兼容旧方法` / `// 兼容旧调用`）
- 新增 API 配置弹窗支持自定义 Base URL 和 Model，全面兼容 OpenAI 格式接口
- 命名卡片/更多菜单等弹窗使用事件委托避免重复绑定

**涉及文件**：`js/ui.js`

### 🐛 Bug #8：推荐话题点击后只填充输入框不发送 + Enter 键无法发送（2026-06-12）

| 项目 | 详情 |
|------|------|
| **现象** | 用户在聊天页点击推荐话题 chip，内容被填入输入框但不会自动发送；按 Enter 键同样不响应。需手动点击发送按钮才能发消息 |
| **原因** | `this.appState.isLoading` 在持久化时被保存为 `true`（上一次会话中途崩溃/关闭导致），下次 `loadData()` 恢复状态后 `isLoading` 泄漏为新会话的阻塞锁。`handleChatSend()` 第一行 `if (this.appState.isLoading) return` 静默返回，导致 chip 只填了 `input.value` 但发送被拦截，Enter 键同理 |
| **泄漏路径** | `handleChatSend()` 设 `isLoading=true` → 页面关闭/崩溃 → `beforeunload` 触发 `saveData()` → `isLoading: true` 写入 localStorage → 下次加载恢复为 `true` → 永久阻塞 |
| **修复** | **三层防御**：① `loadData()` 恢复 appState 后强制 `isLoading=false`、`isEnteringChat=false`（启动层）；② `openChatFromHome()` 入口强制清零（每次进聊天时释放）；③ `handleChatSend()` 用外层 `try-finally` 统一释放锁，覆盖离线匹配 return、API 成功、API 异常、同步崩溃四种路径。同时 `chatInput.value=''` 提升到函数入口统一清空，避免多处重复 |
| **涉及文件** | `js/app.js` |
| **状态** | ✅ 已修复 |

### 🐛 Bug #11：装扮按钮无法打开装扮弹窗（2026-06-13）

| 项目 | 详情 |
|------|------|
| **现象** | 点击「🎨 装扮」按钮，装扮弹窗不显示，无报错 |
| **原因** | `ui.js` 的 `ensureCleanState()` 每次页面切换时给 `#dressup-overlay` 设置 `style.display='none'`。此后 `openDressUp()` 仅添加 CSS class `.active { display: flex }`，但 inline style 优先级高于 CSS class，`display:none` 覆盖了 `display:flex`，弹窗始终不可见 |

### 变更 #8：企鹅名字全链路动态化 + 聊天段落渲染（2026-06-12）

本轮改动解决两个用户体验痛点：**企鹅名字在不同系统中不一致**，以及 **AI 回答不分段导致阅读困难**。

#### 8.1 企鹅名字全链路动态化

**问题**：企鹅名字在三个系统中各自为政——System Prompt 里写死"蛋蛋/波波/酷酷/飞飞"，离线问答库写死"一路有鹅"，只有 UI 顶部显示用户自定义名字。用户取名后，AI 聊天和离线回答仍用固定名字。

**改动**：

| 模块 | 改动 | 文件 |
|------|------|------|
| System Prompt | 四个年级开场句 `名叫"蛋蛋"` → `名叫"{penguinName}"`，使用占位符统一 | `js/chat.js` |
| `buildSystemPrompt` | 增加 `{penguinName}` → `penguinData.name` 替换 + 动态上下文中直接用 `penguinName` | `js/chat.js` |

### 变更 #16：内容哈希缓存破坏机制（2026-06-13）

**背景**：多次部署后 CDN 边缘节点仍缓存旧版本 JS/CSS 文件，即使源站已更新。传统 CDN 刷新 API 无法通过工具调用，导致云端始终是旧版本。

**方案**：对所有 JS（19 个）和 CSS（1 个）文件计算 SHA256 内容哈希，生成带哈希的新文件名（如 `app.2646a0a7.js`），更新 `index.html` 引用。由于新文件名完全不同，CDN 无任何缓存，**首次请求即命中最新内容**。

| 项目 | 说明 |
|------|------|
| JS 文件哈希化 | 19 个 → `js/basename.xxxxxxxx.js` |
| CSS 文件哈希化 | 1 个 → `css/style.77ed5e6a.css` |

## 九、开发心得与关键决策

### 9.1 关键架构决策

1. **纯前端架构（无后端）**：选择 Vanilla JS + localStorage，降低部署门槛，用户只需浏览器即可运行。
2. **Canvas 而非 DOM 渲染企鹅**：企鹅外观细节丰富，Canvas 2D 绘图更灵活，支持逐帧动画。
3. **三级匹配策略（离线→API→兜底）**：保障用户在无网络/无 API Key 时仍可使用。
4. **数据结构提前预留**：`completedActivities`、`dailyActivities`、`unlockedBadges` 等字段在 v1 就已定义，为后续 P2 功能落地做好准备。

### 9.2 踩过的坑

1. **Canvas 在隐藏元素中尺寸计算**：`getBoundingClientRect()` 返回 0，需要用 `canvas.width` 属性兜底。
2. **`requestAnimationFrame` 的帧率差异**：不同设备帧率不同（30/60/120fps），动画进度不能依赖帧数，需要基于时间或帧数兜底。
3. **localStorage 的隐性限制**：不仅有大小的 Quota 限制，还有写入频率限制，需要异常覆盖。
4. **大量 JS 文件的加载顺序**：依赖关系必须严格（storage → keywords → activity → offline-qa → memory → diary → ... → app），顺序错误会导致运行时 `undefined`。
